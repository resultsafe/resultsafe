# Documentation Governance Standard

## 1. Назначение

Настоящий стандарт закрепляет правила управления документацией проекта как единой системой знаний.

Цели стандарта:

- обеспечить управляемую, долгоживущую и масштабируемую систему документации;
- исключить множественные и конфликтующие источники истины;
- повысить пригодность документации для сопровождения, эксплуатации, онбординга, RAG и docs portal;
- определить роли контуров `/docs/` и `/site/docusaurus/`;
- зафиксировать требования к актуальности, ownership, lifecycle и включению материалов в knowledge corpus.

---

## 2. Область действия

Стандарт распространяется на:

- всю документацию проекта;
- архитектурные документы;
- описания сервисов, модулей и библиотек;
- API и контракты;
- runbook, troubleshooting и operational docs;
- onboarding-материалы;
- glossary;
- ADR, RFC и policy-документы;
- контуры `/docs/` и `/site/docusaurus/`;
- материалы, претендующие на включение в RAG-корпус.

---

## 3. Основные принципы

### 3.1 Документация это система знаний

Документация рассматривается не как набор Markdown-файлов, а как управляемая система знаний проекта.

### 3.2 Наличие документа не равно качеству знания

При конфликте между "формально документ есть" и "по нему реально нельзя надежно понять и сопровождать систему" выбирается второе, а качество документа считается недостаточным.

### 3.3 Код не заменяет документацию

Наличие исходного кода не считается заменой архитектурной, эксплуатационной, доменной и onboarding-документации.

### 3.4 У критичных знаний должен быть source of truth

Для каждого критичного знания должен быть явно определен единый источник истины.

### 3.5 Documentation system должна быть устойчивой к росту

Структура документации должна сохранять понятность и управляемость при росте числа сервисов, команд, доменов и operational-практик.

---

## 4. Роли контуров документации

### 4.1 `/docs/` - master knowledge layer

`/docs/` является основной базой знаний репозитория.

Назначение:

- хранение полной и рабочей базы знаний проекта;
- фиксация архитектурных, технических, операционных и доменных знаний;
- хранение source-oriented документации;
- поддержка сопровождения, эксплуатации, RAG и внутреннего онбординга.

Свойства:

- может содержать более полные, глубокие и внутренние материалы;
- может включать исследовательские, рабочие и промежуточные знания;
- является primary knowledge base проекта;
- является основным кандидатом в source corpus для RAG, если явно не указано иное.

### 4.2 `/site/docusaurus/` - publication layer

`/site/docusaurus/` является публикуемым документационным контуром проекта.

Назначение:

- предоставление отобранной, нормализованной и навигационно оформленной документации;
- публикация human-facing документации;
- формирование docs portal для команды и новых участников;
- представление curated knowledge layer на основе основной базы знаний.

Свойства:

- должен быть чище, стабильнее и навигационно понятнее, чем `/docs/`;
- не обязан содержать весь объем внутренних рабочих материалов;
- должен содержать только те знания, которые готовы к публикации и сопровождению как docs portal;
- является publication layer, а не полной копией `/docs/`.

### 4.3 Специальные слои внутри `/docs/`

- `docs/archive/` - legacy/historical layer (не current, не authoritative source).
- `docs/_generated/` - derived machine-generated layer (не authored source-of-truth).
- `docs/_attachments/` и `docs/_templates/` - вспомогательные assets/tooling слои.

---

## 5. Обязательное разграничение ролей

В проекте закрепляется следующее разграничение:

- `/docs/` - master knowledge layer;
- `/site/docusaurus/` - publication layer.

Это означает:

- не вся информация из `/docs/` обязана публиковаться в `/site/docusaurus/`;
- все критичные знания должны иметь явно определенный source of truth;
- `/site/docusaurus/` не должен превращаться в хаотичную копию `/docs/`;
- `/docs/` не должен использоваться как неструктурированный архив без правил актуальности и ownership.

---

## 6. Source of Truth

### 6.1 Обязательность source of truth

Для каждого критичного знания должен быть явно определен source of truth.

К критичным знаниям относятся:

- архитектура;
- структура монорепозитория;
- сервисы и приложения;
- shared libraries и packages;
- API и контракты;
- модели данных;
- бизнес-логика;
- конфигурация и окружения;
- build / test / CI / CD;
- инфраструктура и deployment;
- observability;
- security;
- runbook / operations / troubleshooting;
- onboarding;
- ADR и архитектурные решения.

### 6.2 Нормативное правило

Если знание критично для сопровождения, эксплуатации, архитектуры, онбординга или RAG, должен быть явно определен его source of truth:

- по умолчанию — `/docs/` (или код/конфиги для runtime-контрактов);
- `/site/docusaurus/` допускается только как projection/publication-слой;
- исключение (portal-page как source-of-truth на узкий scope) допускается только через явную запись в override-реестре.

### 6.3 Дублирование source of truth запрещено

Не допускается наличие двух равноправных и расходящихся главных версий одного и того же знания.

### 6.4 Реестр исключений source-of-truth

Канонический реестр исключений:

- `docs/_generated/identifier-registry/DOC-PORTAL-SOURCE-OVERRIDES.json`.

Для каждого исключения обязательны:

- область действия (scope);
- owner;
- первичный документ;
- производный документ;
- правило синхронизации.

---

## 7. Правило синхронизации между контурами

Если материал существует и в `/docs/`, и в `/site/docusaurus/`, должно быть явно определено:

- какой документ является первичным;
- какой документ является опубликованной, адаптированной или производной версией;
- кто отвечает за актуальность;
- когда и при каких изменениях требуется синхронизация.

Запрещается:

- держать две равноправные, но расходящиеся версии одного и того же знания;
- публиковать в `/site/docusaurus/` материал, конфликтующий с `/docs/`, без явной пометки;
- использовать оба контура как конкурирующие главные базы знаний.

---

## 8. Требования к качеству документа

Документ считается качественным не по факту существования, а по факту практической пригодности.

Документ не считается достаточным, если:

- он не позволяет надежно понять систему;
- он не позволяет безопасно сопровождать или изменять компонент;
- он не позволяет выполнять эксплуатационные действия;
- он требует обязательного чтения исходного кода как единственного реального источника истины;
- он слишком фрагментарен, расплывчат или устарел;
- его роль в knowledge system неочевидна.

---

## 9. Lifecycle документации

Вся документация должна быть явно разделена как минимум на следующие состояния:

- `current` - актуальная рабочая документация;
- `legacy` - документация по устаревающим, но еще значимым частям;
- `archive` - исторические материалы, не являющиеся рабочим источником истины.

Обязательные требования:

- должно быть понятно, что является актуальным;
- должно быть понятно, что является устаревающим;
- должно быть понятно, что является архивным;
- должно быть явно видно, что не должно использоваться как рабочий source of truth.

Нормативная реализация lifecycle в документах:

- workflow-статус фиксируется в `status`;
- уровень пригодности базы знаний фиксируется в `kb_lifecycle` (`current`/`legacy`/`archive`);
- официальный маппинг `status -> kb_lifecycle` задается `DOC-004`.

Запрещается:

- смешивать `current` и `archive` без маркировки;
- оставлять `legacy`-документы в рабочем контуре без явной пометки;
- допускать попадание `archive`-контента в активный RAG-корпус без специального правила.

---

## 10. Ownership

Для критичных разделов документации должен быть определен owner или ответственная зона.

Нормативная реализация ownership:

- `owner` или `owners` в frontmatter;
- контракт полей и обязательность по типам документов задаются `DOC-004`.

Минимально ownership обязателен для:

- архитектуры;
- API и контрактов;
- инфраструктуры;
- deployment;
- runbook;
- security;
- onboarding;
- glossary;
- правил работы с knowledge base.

Если у критичного знания нет owner, это считается дефектом зрелости knowledge base.

---

## 11. Governance rule для документов без роли

Любой документ, не имеющий понятной роли, актуальности, owner и места в lifecycle документации, считается кандидатом на архивирование, переработку или исключение из knowledge corpus.

Это правило применяется в том числе к:

- черновикам без статуса;
- дублирующим заметкам;
- abandoned docs;
- документам без понятной аудитории;
- документам без признаков актуальности;
- материалам, чья ценность для сопровождения, онбординга, эксплуатации или RAG неочевидна.

Для терминов и типов документов:

- `RFC`, `policy`, `glossary` нормируются в `DOC-004` (префиксы, типы, размещение);
- `abandoned docs` трактуются как документы без owner/актуальности и с `kb_lifecycle != current`.

---

## 12. Правило минимального дублирования

Дублирование допускается только если оно:

- осознанно;
- ограничено;
- не конфликтует;
- не создает несколько равноправных source of truth;
- помогает разным сценариям потребления знания.

Допустимое дублирование:

- краткий overview в Docusaurus и более полный reference в Obsidian;
- краткая публикационная версия и полная рабочая версия;
- summary-слой для onboarding и глубокий инженерный слой для сопровождения.

Недопустимое дублирование:

- две расходящиеся главные версии одной темы;
- несколько документов с одинаковым назначением и разным смыслом;
- копии страниц без правила актуализации;
- дубли, ухудшающие поиск, RAG и понимание.

---

## 13. Правило включения в RAG

По умолчанию:

- `/docs/` рассматривается как основной кандидат в source corpus для RAG;
- `/site/docusaurus/` подключается как дополнительный curated tier по explicit профилю retrieval.

### 13.1 В индекс можно включать

- актуальные документы;
- самодостаточные документы;
- reference-материалы;
- runbook;
- ADR;
- glossary;
- архитектурные описания;
- API и контракты;
- инструкции и troubleshooting, если они актуальны.

### 13.2 В индекс нельзя включать без специального решения

- `archive`;
- abandoned docs;
- временные заметки;
- черновики;
- шумные дубли;
- autogenerated-файлы без knowledge value;
- материалы с неясной актуальностью;
- конфликтующие версии одного и того же знания.

### 13.3 Нормативное правило

RAG-индекс должен строиться не из всех файлов подряд, а только из управляемого и отфильтрованного knowledge corpus.
Канонический default-профиль и опциональные профили retrieval задаются `SPEC-014` и `RAG-CORPUS-PROFILE.json`.

### 13.4 Source vs retrieval runtime rule

Для knowledge base обязательно:

- authored current canonical docs остаются primary source of truth;
- `PostgreSQL + pgvector` слой используется только как secondary processing/retrieval runtime;
- chunks/embeddings/vector indexes/generated registries/retrieval metadata считаются derived operational artifacts;
- default retrieval обязан использовать metadata-first filtering и не может включать legacy/archive/noise по умолчанию.

---

## 14. Правило публикации в Docusaurus

В `/site/docusaurus/` должны попадать только материалы, которые:

- понятны человеку;
- навигационно оформлены;
- имеют устойчивую структуру;
- готовы к публикации;
- не конфликтуют с source of truth;
- поддерживаются как часть docs portal.

### 14.1 В `/site/docusaurus/` обязательно должны быть представлены

- overview проекта;
- архитектурный обзор;
- основные сервисы и модули;
- onboarding;
- ключевые how-to;
- reference по важным интерфейсам;
- runbook / troubleshooting для основных сценариев;
- glossary, если доменная область сложная.

### 14.2 В `/site/docusaurus/` не должны попадать без необходимости

- сырые рабочие заметки;
- внутренние черновики;
- шумные промежуточные материалы;
- слабо структурированные исследования;
- дубли без навигационной ценности.

---

## 15. Информационная архитектура

Документация должна быть организована как система знаний, а не как случайный набор Markdown-файлов.

Должны быть различимы следующие типы материалов:

- overview;
- reference;
- how-to;
- runbook;
- ADR;
- onboarding;
- troubleshooting;
- glossary;
- policy.

Смешение разных типов документов без явной структуры считается дефектом информационной архитектуры.

---

## 16. Актуальность и доверие

Для документа должно быть понятно:

- актуален ли он;
- является ли он нормативным или обзорным;
- можно ли использовать его как рабочий источник истины;
- является ли он `current`, `legacy` или `archive`.

Документы без ясных признаков актуальности и статуса снижают доверие ко всей knowledge system.

---

## 17. Недокументированное знание

Любой критичный operational, architectural или domain knowledge, существующий только:

- в головах команды,
- в коде,
- в случайных заметках,
- в устной передаче знаний,

считается недокументированным.

---

## 18. Минимальные требования зрелости knowledge base

Knowledge base считается зрелой только если:

- критичные знания покрыты документированно;
- source of truth определен;
- ownership существует;
- `current` / `legacy` / `archive` различимы;
- дублирование контролируемо;
- мусор и шум не засоряют рабочий контур;
- документация пригодна для сопровождения, онбординга и эксплуатации;
- существует управляемое правило включения материалов в RAG;
- publication layer не конфликтует с master knowledge layer.

---

## 19. Признаки дефекта зрелости

Ниже перечислены признаки незрелой documentation system:

- несколько конфликтующих документов по одной теме;
- отсутствие source of truth;
- отсутствие ownership;
- отсутствие маркировки `current` / `legacy` / `archive`;
- docs portal как хаотичная копия внутренней базы знаний;
- критичные знания, спрятанные в случайных README;
- operational knowledge, существующий только в головах команды;
- `archive` и `legacy`, попадающие в активный RAG-контур;
- документы без понятной роли, аудитории и lifecycle;
- онбординг, эксплуатация или архитектурное понимание, требующие чтения кода как primary source.

---

## 20. Правила применения стандарта в аудите

При аудите documentation system необходимо руководствоваться следующими правилами:

- не путать "документ существует" и "документ реально полезен";
- не компенсировать слабость documentation system знанием из кода;
- считать отсутствие source of truth серьезным дефектом;
- считать смешение `current` / `legacy` / `archive` серьезным дефектом;
- считать отсутствие ownership дефектом зрелости;
- отдельно оценивать master knowledge layer и publication layer;
- отдельно выявлять gaps между `/docs/` и `/site/docusaurus/`.

---

## 21. Короткая нормативная версия

### 21.1 Базовые правила

1. `/docs/` - master knowledge layer: основная, полная, рабочая база знаний репозитория; primary knowledge base; основной кандидат в source corpus для RAG.
2. `/site/docusaurus/` - publication layer: отобранная, нормализованная, навигационно оформленная, human-facing документация; curated docs portal.
3. Для каждого критичного знания должен быть явно определен source of truth:
   - по умолчанию `/docs/` (или код/конфиги для runtime-контрактов),
   - `/site/docusaurus/` только как projection-слой,
   - исключения только через override-реестр с owner и sync-правилом.
4. Если материал существует в обоих контурах, обязательно должно быть определено:
   - какой документ первичен;
   - какой документ производный;
   - кто отвечает за актуальность;
   - как поддерживается синхронизация.
5. При конфликте между "формально документ есть" и "по нему реально нельзя надежно понять и сопровождать систему" выбирается второе, а качество документа считается недостаточным.
6. Вся документация должна быть явно разделена на:
   - `current`
   - `legacy`
   - `archive`
7. В активный RAG-индекс не должны попадать:
   - `archive`
   - abandoned docs
   - временные заметки
   - черновики
   - шумные дубли
   - autogenerated-файлы без knowledge value
   - материалы с неясной актуальностью
8. `/site/docusaurus/` не должен быть полной копией `/docs/`. Он должен быть curated publication layer, а не дублирующим хаотичным контуром; включается в retrieval только по explicit профилю.
9. Для критичных разделов документации должен быть определен owner или ответственная зона.
10. Дублирование допускается только как контролируемое и не конфликтующее. Множественные source of truth не допускаются.
11. Любой документ, не имеющий понятной роли, актуальности, owner и места в lifecycle документации, считается кандидатом на архивирование, переработку или исключение из knowledge corpus.

### 21.2 Жесткое правило governance

Любой документ, не имеющий понятной роли, актуальности, owner и места в lifecycle документации, считается кандидатом на архивирование, переработку или исключение из knowledge corpus.

### 21.3 Жесткое правило полноты знаний

Любой критичный operational, architectural или domain knowledge, существующий только в головах команды, в коде или в случайных заметках, считается недокументированным.

## 23. Финальная нормативная формулировка

В проекте documentation system должна поддерживаться как управляемая, актуальная и масштабируемая система знаний.

Любой документ обязан иметь:

- понятную роль;
- понятную аудиторию;
- понятный статус актуальности;
- owner или ответственную зону;
- место в lifecycle документации;
- понятное отношение к source of truth.

Документы, не соответствующие этим требованиям, не должны по умолчанию считаться частью надежного knowledge corpus и подлежат переработке, архивированию или исключению.
