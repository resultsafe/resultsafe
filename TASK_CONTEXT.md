## 1. Назначение шаблона

[↑ к оглавлению](#tc-toc)

`TASK_CONTEXT.md` нужен для конкретной задачи, чтобы:
- ИИ не делал полный рескан репозитория;
- все решения по задаче были в одном месте;
- handoff другому инженеру/агенту был без потери контекста.

Правило использования:
- на каждую значимую задачу создавать копию этого шаблона как `TASK-<short-id>.md` в `docs/obsidian/tasks/{backlog|active|blocked|done}` (или в корне только как временный local draft);
- заполнять разделы 3–11 до начала реализации и обновлять по ходу.

## 2. Быстрый старт по задаче

[↑ к оглавлению](#tc-toc)

Минимальный порядок чтения перед работой:
1. `TASK-<id>.md` (экземпляр этого шаблона, заполненный под задачу).
2. `AI_CONTEXT.md` (общие правила проекта).
3. Целевые файлы из раздела 5 текущей задачи.

Минимальный порядок действий:
1. Подтвердить границы и acceptance criteria.
2. Сделать изменение в целевом scope.
3. Запустить проверки из раздела 8.
4. Обновить журнал изменений и handoff.

<a id="tc-8"></a>

## 8. Проверки и команды

[↑ к оглавлению](#tc-toc)

Базовый набор:

```bash
pnpm install
pnpm run workspace:list
pnpm run smoke:docs
pnpm run smoke:docs:report
pnpm run docs:verify
pnpm run docs:sync-registries
pnpm run docs:sync-identifiers
pnpm run docs:sync-semantic-modules
pnpm run docs:sync-python-parity
pnpm run docs:sync-entity-catalog
pnpm run automation:test
```

`pnpm lint/test/build` в root считать non-canonical до завершения migration-потока пакетных конфигов; для качества использовать package-targeted запуск:

```bash
pnpm --filter <package-name> run <script>
```

Локальный набор для задачи:

```bash
<добавь точечные команды для целевого пакета>
```

Ожидаемый результат:
- команды завершаются без ошибок;
- если есть известный флейк, фиксировать явно с причиной.

Режим запуска shell-команд для агента:
- запускать команды только в non-interactive режиме;
- для automation-задач использовать канонический Python CLI:
  `python -m tools.automation <group> <command> ...`;
- если команда требует TTY/stdin-terminal, не запускать её в таком виде;
- подбирать неинтерактивный эквивалент (например, `codex exec ...` вместо интерактивного `codex ...`).

## 10. Handoff формат

[↑ к оглавлению](#tc-toc)

Итоговый handoff должен содержать:
1. Что сделано (по пунктам).
2. Какие файлы изменены.
3. Что проверено (команды + итог).
4. Что не удалось/ограничения.
5. Следующие шаги (если нужны).

Краткий шаблон:

```text
Done:
- ...

Changed:
- path/to/file

Verified:
- <command> -> OK/FAIL

Limitations:
- ...

Next:
1. ...
```
