# Инженерные стандарты для каталога `tools/`

Этот документ — каноническое описание того, как мы проектируем и документируем утилиты внутри каталога `tools/`. Он дополняет правила из `docs/RULES.md` и `docs/specs/DOC-010-package-and-publishing-structure-canonical-rule.md`.

## 1. Область действия

- Применяется ко всем подкаталогам в `tools/`, включая новые утилиты.
- Любые отступления фиксируются в README конкретного инструмента.

## 2. Формат ответов (Rust-style Result)

- Все CLI-утилиты в `tools/` выводят JSON в стиле Rust `Result`.
- Успех:
  ```json
  {"ok": true, "value": <payload>}
  ```
- Ошибка:
  ```json
  {"ok": false, "error": {"code": "string", "message": "human readable", "details": {...?}}}
  ```
- Стандартизация вывода:
  - `stdout` — только итоговый JSON.
  - `stderr` — служебные сообщения, логирование, трассировки.
  - Код выхода: `0` — успех, `1` — предсказуемые ошибки валидации/входа, `2` — непредвиденная внутренняя ошибка.

## 3. Структура каталога инструмента

```
tools/<tool_name>/
  README.md                краткое описание и примеры запуска (ru)
  docs/
    API.md                 контракт ввода/вывода (включая код выхода)
    USAGE.md               сценарии использования, аргументы CLI
    CHANGELOG.md           (опционально) история изменений
    AI_RULES_PROMPT.md     (опционально) промпт для генерации правил ИИ
  src/ или корневые .py    основная реализация
  tests/                   автотесты (unittest/pytest)
  data/                    (опционально) образцы/фикстуры, не код
  __init__.py              по необходимости
```

Минимальный набор файлов: `README.md`, основной скрипт, хоть один тестовый файл.

## 4. Нейминг и стиль кода

- Python 3.11+, стиль PEP8; имена файлов — `snake_case`.
- Локализация комментариев и docs — русский, если иное не требуется внешним пользователям.
- Избегать сторонних зависимостей; если нужны — фиксировать версии и описывать в `README.md`.

## 5. Требования к CLI

- Аргументы и флаги документируются в `docs/USAGE.md`.
- Скрипты должны поддерживать `--help`.
- Не допускается интерактивный ввод; все параметры через флаги.
- Логи — в stderr; JSON-результат — в stdout.

## 6. Тестирование

- Обязательно: модульные тесты ключевой логики (в `tests/`).
- Рекомендуется: `python -m compileall tools/<tool_name>` как минимальная проверка.

## 7. Документация и ссылки

- Каждый инструмент ссылается на релевантные канонические документы: `docs/RULES.md`, `DOC-010`, профильные SPEC при необходимости.
- При добавлении нового инструмента дополняйте список в `tools/README.md` (см. ниже).

## 8. Реестр ID для вызова из ИИ

- Каждый инструмент получает стабильный `tool_id` (строка в `kebab-case`), числовой `tool_code` (натуральное число) и `tool_uuid` (UUID v4).
- Реестр ведётся в `tools/README.md` в виде таблицы: `tool_code`, `tool_id`, `tool_uuid`, `path`, `description`, `version`, `status`.
- Версия — семвер `MAJOR.MINOR.PATCH`, обновляется при изменении контракта ввода/вывода или поведения.
- Принципы:
  - Идентификаторы неизменяемы. Переименование инструмента => новый набор `tool_code`/`tool_id`/`tool_uuid`, старый помечается как `deprecated`.
  - Один `tool_id` — один исполняемый вход (CLI). Если есть подкоманды, документируйте их в `docs/USAGE.md`.
  - Для ИИ-агентов используйте `tool_id` или `tool_code` как ключ вызова; `tool_uuid` служит для глобальной корреляции и трассировки.
- Минимальный набор данных в таблице: `tool_code`, `tool_id`, `tool_uuid`, `path`, `description`, `version`, `status` (`active`|`deprecated`).

## 9. Хранение исходников

- Все стандарты для `tools/` хранятся в этом файле (`tools/STANDARDS.md`). Он является источником правды для разработчиков внутренних утилит.
