# resultsafe — Domain Model and Glossary

> Single source of truth for all domain terminology.
> All documents, code, tests, and contracts use these terms exactly as defined.
> Ubiquitous language: one concept = one term — no synonyms in use.
>
> Before introducing a new term anywhere: check this file.
> If missing: add here first, then use in other documents.

---

## Core domain terms

### Result

A discriminated union type representing the outcome of an operation that
can either succeed or fail. A Result is always in exactly one of two states:
`Ok` or `Err`. Never throws — the error path is part of the type signature.

- **Ok**: the success variant — carries a value
- **Err**: the failure variant — carries an error
- **Not a synonym for**: `Either`, `Try`, `Outcome`, `Option`

### Ok

The success variant of a Result. Carries a value of any type.
An Ok result means the operation completed successfully.

- **Not a synonym for**: `Right`, `Success`, `Some`

### Err

The failure variant of a Result. Carries an error of any type.
An Err result means the operation failed.

- **Not a synonym for**: `Left`, `Failure`, `None`, `Error` (the class)

### Value

The inner payload of an Ok result. Accessible only after narrowing
(confirming the Result is Ok).

### Error

The inner payload of an Err result. Accessible only after narrowing
(confirming the Result is Err). Note: `error` is a domain term, not
necessarily a JavaScript `Error` instance.

### TaskResult

An asynchronous Result — a `Promise<Result<T, E>>`. Represents a computation
that will produce a Result in the future.

- **Not a synonym for**: `AsyncResult`, `FutureResult`, `PromiseResult`

### Codec

An encoder/decoder pair that transforms values between two representations,
using Result-based error handling. A Codec consists of an `encode` function
and a `decode` function.

### Pipeline

A chain of Result-returning operations composed via `andThen`, `map`,
or `orElse`. The error propagates automatically through the chain without
explicit checking at each step.

### Refiner

A function or combinator that transforms an untyped Err into a discriminated
union of typed error variants. Used to narrow the error type from a generic
`unknown` to specific, named variants.

### Variant

A named, typed subtype within a discriminated union of errors. Used in
`refineResult` and `matchVariant` to identify and handle specific error cases.

### RefinedResult

A Result whose Err type has been narrowed to a specific discriminated union
via a refiner. Each Err variant is individually typed and handleable.

### Ubiquitous language

The practice of using the same terms in domain documents, code identifiers,
test descriptions, and contracts. No synonyms, no paraphrasing in technical
contexts.

## Documentation domain terms

### Contract

A language-neutral specification of the observable behavior of a single API
symbol — written without any programming language syntax. Describes what,
never how.

### Language-neutral

A description that contains no syntax, idioms, or constructs specific to
any programming language. Can be implemented identically in TypeScript,
Python, Go, and any other language.

### Parity

Two implementations satisfy the same contract identically. Full parity means
no behavioral differences between language implementations.

### Parity deviation

A known, documented behavioral or syntactic difference between an
implementation and its contract. Deviations are deliberate — never silent.

### Source of truth

The single document or file that is authoritative for a given piece of
knowledge. All other references to that knowledge link to the source of
truth and do not copy it.

### Authored layer

Documents written by humans or AI agents. Editable. Primary source of truth.

### Derived layer

Documents generated by scripts from authored sources. Not editable manually.
Not committed to git.

### Legacy layer

Archived documents. Read-only. Not authoritative.

## Testing domain terms

### Red phase

The TDD cycle step where a failing test is written. No production code exists
yet for the behavior being tested. The test must fail for the right reason.

### Green phase

The TDD cycle step where minimum production code is written to make the
failing test pass. No more code than necessary.

### Refactor phase

The TDD cycle step where code is improved — extracting functions, improving
naming, eliminating duplication — without changing observable behavior.
All tests must remain passing.

### Type test

A compile-time assertion that a TypeScript type has the exact shape expected.
Written using `expect-type` or `tsd`. Verifies that generic constraints,
discriminated union narrowing, and return types are correct.

### Contract compliance test

A test that directly verifies a specific clause from a language-neutral
contract. References the contract ID in a comment.
Example: `// Verifies: SPEC-013 §Behavior: Case result is Ok(value)`

### Export integrity test

A test that verifies the complete public API surface of a package has not
accidentally changed. Located in `__tests__/unit/exports/index-exports.test.ts`.

| Attribute | Value |

## Business rules

- A freed package ID (PACKAGE-XXXX) is never reused
- A freed document ID (SPEC-NNN, ADR-NNN, etc.) is never reused within its scope
- Breaking API changes require a semver major bump and a new ADR
- Every exported symbol must have a TSDoc comment before a release
- Every exported symbol must have contract compliance tests
- Parity deviations are always documented — never silent
